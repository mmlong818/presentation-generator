# 版式模板设定规则（PPT Layout Spec）

> 这份文档定义"什么是一个合格的版式模板"。
> 你按这个规则设计新版式，把成果交给开发者落地（或丢给 AI 让它直接写代码）。

---

## 一、版式的三个层次

每个版式必须同时定义三件事，缺一不可：

| 层次 | 内容 | 示例 |
|---|---|---|
| **1. 信息功能** | 这张幻灯片解决什么演讲表达需求？ | "对比 A 和 B 的差异" |
| **2. 数据契约** | 这张幻灯片需要哪些字段？字段类型？ | `{ heading, left:{title,items[]}, right:{title,items[]} }` |
| **3. 视觉构成** | 字段怎么排在 1920×1080 画布上？ | "左右两列均分，左暗右亮 accent 色" |

只有功能没数据契约 → AI 不知道怎么填。
只有数据契约没视觉 → 拿到数据画不出来。
三者齐备才是一个完整版式。

---

- brutalist-mono: ________________

（给 AI 看的 few-shot，必须填写真实可用的示例）

  "type": "xxx",

- [ ] 没有"圆角卡片+左侧彩色竖条"（AI slop 模板）

## 三、必须遵守的硬性规则

### 数据契约层

1. **字段命名**：camelCase，英文。
2. **必填 vs 可选**：少即是多。eyebrow/source/desc 这类装饰字段全部用 `?` 可选，AI 会决定要不要填。
3. **数组类字段限上下界**：`steps: [3, 5]`、`stats: [1, 3]`。AI 倾向凑满，不限会塞太多。
4. **highlight 字段约定**：如果版式支持高亮某个词，统一用 `highlight?: string`（单个）或 `highlight?: string[]`（多个）。
5. **诚信占位约定**：数据类字段（数字、引用来源）必须支持占位语法：`value: "—"`，`source: "需引用：[来源类型]"`。

### 视觉层

6. **画布固定 1920×1080**，不可滚动。竖向预算计算公式：
   ```
   总高度 = 上 padding + Σ(font_size × line_height × 行数) + Σ 元素间距 + 下 padding
   ≤ 1080
   ```
7. **字号下限 22px**，再小投影仪看不清。
8. **不许用 emoji 当装饰**。要图标就用 SVG。
9. **不许出现"圆角卡片 + 左边彩色竖条 + 三列"** —— 这是 AI 套路化最严重的样式。
10. **5 主题适配**：版式必须考虑暗模式（tech-utility）会把白底卡片翻车、纯黑白模式（brutalist）会把渐变翻车。

### 内容层

11. **标题必须是观点句**，不是话题词（这是给 AI 的提示，但版式的字段命名也要引导：用 `argument` 不用 `point`，用 `heading` 不用 `title` 来暗示要写完整观点）。
12. **每条 item / point 限定**：单行不可换行（≤ 25 字中文 / 40 字英文）。
13. **整篇 deck 同类型 ≤ N 张**：比如 statement（单句冲击）≤ 3 张，避免疲劳。

## 四、版式分类（避免重复定义）

新版式应能归入或扩展以下大类，否则要慎重：

| 大类 | 已有版式 | 缺什么 |
|---|---|---|
| **开场 / 收尾** | cover, cta, checklist | 也许：thanks（致谢）、qr（扫码） |
| **冲击 / 转折** | statement, quote | 也许：silence（全黑+白字句） |
| **结构 / 流程** | process, timeline | 也许：roadmap（季度路线图）、loop（循环图） |
| **数据 / 证据** | data | 也许：chart（柱状/折线，需 SVG 实现）、table（小表格） |
| **对比 / 关系** | compare, diagram | 也许：matrix（2×2 矩阵）、quadrant（象限图）、stack（栈式对比） |
| **论证 / 推理** | argument | 也许：syllogism（三段论）、causality（因果链） |
| **故事 / 人物** | quote | 也许：persona（人物肖像+介绍）、case-study（案例） |
| **行动 / 决策** | cta | 也许：question（开放式提问留给观众）、poll（互动投票） |

**新版式提议优先级排序**：
1. 当前已遇到但被迫硬塞进 argument/compare 的需求 → 高
2. 用户场景明确（如金融汇报需要 KPI 看板）→ 高
3. 视觉炫但功能可被现有版式覆盖 → 低
4. 只为了"看起来花样多" → 拒绝

---

## 五、跨主题适配的 6 个常见坑

设计新版式必须脑内过一遍：

1. **暗模式（tech-utility）**：白底卡片 → 翻车。要用 `t.paper`（自动深色）。
2. **纯黑白（brutalist-mono）**：渐变 → 翻车。任何 `linear-gradient` 必须有降级。
3. **衬线（editorial-monocle / soft-warm）**：等宽数字会出戏。数字大字号要测试衬线效果。
4. **窄 accent 色**：`accent` 只用于"有限的强调"，整页都用 accent 会破坏所有主题。
5. **页码刊头**：tech-utility/modern-minimal/soft-warm 不显示，editorial/brutalist 显示。组件读 `t.showFooter`。
6. **eyebrow 装饰风格**：5 主题各异（普通 / 上线 / 下划线 / 等宽 `>` / 排印帽）。统一用 `<Eyebrow>` 组件，别自己重写。

## 六、提交格式

完成设计后给我一个 `.md` 文件 或 直接在对话中贴出来，包含：

1. **填好的"模板填写表"**（第二节）
2. **简笔画或 ASCII 排版**：让我看到你脑中的画面
3. **3-5 个真实数据示例**：方便我在主题切换下肉眼测试鲁棒性

例如：
> ```
> ┌────────────────────────────────────┐
> │ EYEBROW · 章节眉                    │
> │                                     │
> │  主标题（72px 衬线）                │
> │                                     │
> │  ┌─── 65% ───┐  ┌─── 35% ───┐      │
> │  │ 大数字     │  │ 文字解释   │      │
> │  │           │  │           │      │
> │  └───────────┘  └───────────┘      │
> │                                     │
> └────────────────────────────────────┘
> ```

我拿到后会按你的设计 + 这份规则去落地代码（types / registry / 组件 / 主题适配），通常 15-30 分钟一个。

---

## 七、不要做的事

- ❌ 一次提议 5 个新版式 —— 一次 1-2 个，快速迭代
- ❌ 复制 PowerPoint 现成模板 —— 那是为静态画布设计的，不适合 1920×1080 演讲
- ❌ 设计成"占位框 + 后期填图" —— 我们的版式是文字驱动的，要图请用 `diagram` 占位
- ❌ 字段层级超过 2 层（嵌套对象套数组套对象）—— AI 容易填错
- ❌ 设计需要外部 npm 库才能渲染的版式（动画、图表）—— 暂不支持，用 SVG 解决

## 附录：现有 11 版式速查

| ID | 用途 | 字段密度 | 何时不用 |
|---|---|---|---|
| `cover` | 封面 / 章节起始 | 低 | 中段过渡用别的 |
| `statement` | 一句话冲击 | 极低 | 整篇 ≤ 3 张 |
| `process` | 流程 3-5 步 | 中 | 步骤 > 5 拆页 |
| `data` | 1-3 大数字 | 中 | 数据 > 3 用 chart |
| `compare` | 二元对比 | 中 | 三方对比用 matrix |
| `timeline` | 时间线 | 中 | 无时间属性别用 |
| `argument` | 论点 + 支撑 | 中高 | 论点单一用 statement |
| `quote` | 引言 | 低 | 引言长 > 60 字拆 |
| `diagram` | 图示占位 | 极低 | 内容能用文字别用 |
| `cta` | 行动号召 | 低 | 中段不要用 |
| `checklist` | 总结清单 | 中 | 一句话总结用 cover |

---

**记住三件事**：
1. 数据契约比视觉重要（视觉可改，契约一旦上线难改）。
2. AI 看的是 `whenToUse` + `example`，写得越具体它选得越准。
3. 跨主题先想清楚，不然每次新增版式都要回炉重造 5 遍。
